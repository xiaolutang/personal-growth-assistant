### 核心原则：先看日志，再猜原因

**错误习惯**：根据现象猜测问题位置（"应该是前端状态问题吧"）

**正确做法**：直接查看后端错误日志，往往直接告诉你答案

---

### 调试顺序清单

```
1. 后端日志 → 有没有 500 错误？错误信息是什么？
2. 网络请求 → 请求/响应内容是什么？
3. 前端状态 → 状态是否正确更新？
4. 代码逻辑 → 业务逻辑是否有问题？
```

**口诀**：日志 → 网络 → 状态 → 代码

---

### 多层错误的"洋葱剥皮"现象

**现象**：修复一个问题后，新问题才暴露出来

**原因**：多个错误叠加，前面的错误阻止了后续代码执行

**应对策略**：
- 修复一个问题后，**立即重新测试**，不要假设"应该好了"
- 每次修复后确认：还有其他错误吗？

---

### 可选依赖的优雅处理模式

**场景**：Neo4j、Qdrant、Redis 等服务可能不可用

**错误写法**：预检查属性是否存在
```python
if client and client.connection:  # AttributeError 可能在这里抛出
    client.do_something()
```

**正确写法**：用 try-except 包裹实际操作
```python
if client:
    try:
        client.do_something()
    except Exception as e:
        logging.warning(f"操作失败，忽略: {e}")
```

**原则**：异常要在 try 块里面才能被捕获

### 调试效率对比

| 方法 | 耗时 | 成功率 |
|------|------|--------|
| 先猜原因再验证 | 长 | 低 |
| 先看日志再定位 | 短 | 高 |
