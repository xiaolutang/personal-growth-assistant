## 一、项目定位

### 产品灵魂

**先帮用户把东西记下来，再帮用户把东西连起来，最后帮用户看见自己在成长。**

### 产品定位

**帮助知识工作者沉淀个人思考、推进行动、形成长期成长资产的 AI 个人助手。**

核心差异化：**自然语言交互 + 本地数据主权 + 知识图谱**。用户不需要填表单，直接对话就能创建任务、记笔记、查资料。

### 目标用户定位：知识工作者的小众工具

**定位选择**：B — 「像你一样的人」的小众产品（非纯自用，非大众产品）。

```
目标用户画像：
- 知识工作者 / 技术人
- 有记笔记习惯（或想养成）
- 觉得 Notion 太重、Obsidian 太折腾、ChatGPT 记不住
- 在意数据主权和隐私
- 愿意为好工具付费
```

这是 Obsidian 的早期用户群规模，不做大众化，不和 Notion 拼功能全面，拼「AI 原生 + 数据主权 + 知识图谱」的差异化。

**定位 B 对关键决策的影响**：

| 决策 | 选择 | 理由 |
|------|------|------|
| 移动端 | Flutter MVP（录入优先） | 目标用户需要随时记录，但不需要全功能 |
| MCP 终端接入 | 做 | 技术人群的杀手特性，Claude Code 直接操作 |
| 数据导入 | 不做 | 符合产品哲学，目标用户能接受 |
| 数据导出 | 必须做 | 技术人群最在意数据锁定 |
| 定价 | 开源 + 可选付费 | 参考 Obsidian 模式 |
| Onboarding | 对话式引导 | 用「日知」对话引导，不用傻瓜教程 |
| UI/UX | 需要打磨但不讨好所有人 | 目标用户容忍学习曲线，但不能忍受功能缺失 |

### 竞争分析

| 竞品 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------------|
| Notion | 功能全面、生态丰富 | 云端锁定、AI 弱、速度慢 | 本地数据 + AI 原生交互 |
| Obsidian | 本地优先、插件生态 | AI 是后加的、学习曲线陡 | AI 是一等公民 |
| Todoist | 任务管理极致简单 | 只有任务，没有知识体系 | 任务+笔记+知识图谱一体化 |
| Apple Notes | 零门槛、免费 | 太简单，无法组织知识 | 智能组织 + 语义搜索 |
| ChatGPT | AI 交互能力强 | 没有持久化、没有结构 | AI + 持久化数据 + 知识图谱 |

### 成长模型

产品的「成长」包含三个层次：

```
目标（方向）    → "我要成为全栈工程师"
  ↓ 拆解
习惯（节奏）    → "每周学5小时Rust，每天复盘"
  ↓ 积累
能力（结果）    → "掌握了Rust所有权、生命周期、async"
```

当前产品只有中间层（习惯节奏），缺顶层（目标设定与追踪）和底层（能力度量）。

## 五、架构设计分析

### 5.1 三大核心设计决策

#### 决策一：Markdown 作为 Source of Truth

```
data/users/{user_id}/
├── inbox/inbox-xxx.md      ← 打开就能看
├── tasks/task-xxx.md        ← git 能追踪
├── notes/note-xxx.md        ← 编辑器能改
└── projects/project-xxx.md  ← 迁移成本低
```

| 维度 | 优势 | 不足 |
|------|------|------|
| 数据主权 | 用户拥有数据，不锁定在系统 | 查询性能差，不能做 WHERE/JOIN |
| 可观测性 | 出问题直接 cat 文件 | Markdown 和 SQLite 两份副本需同步 |
| 零迁移 | 换系统拷走 .md 文件即可 | 多进程并发写同一文件会冲突 |
| 版本控制 | 可 git 追踪数据变更历史 | 关联查询需遍历文件或依赖 SQLite |

**本质取舍**：用「存储冗余」换「数据安全」。

#### 决策二：SQLite + Neo4j + Qdrant 三引擎并行

```
Markdown ──同步──→ SQLite ──同步──→ Neo4j + Qdrant
  (主源)            (索引)          (图谱)   (向量)
```

| 查询需求 | 引擎 | 原因 |
|---------|------|------|
| 精确过滤：status=doing AND date=today | SQLite | 结构化过滤是关系数据库本行 |
| 概念关联：React 和哪些知识有关 | Neo4j | 关系查询是图数据库本行 |
| 语义搜索：关于前端性能优化的笔记 | Qdrant | 向量相似度能理解「意思」 |

**优势**：查询质量高（混合搜索 向量70% + 全文30%）、各司其职、知识图谱是真正差异化。

**不足**：同步复杂度高、运维成本高（4个服务）、开发心智负担大、存在数据不一致窗口。

#### 决策三：LangGraph + SSE 流式对话

| 对比 | 表单模式 | 对话模式 |
|------|---------|---------|
| 创建任务 | 打开→填标题→选状态→选日期→提交 | "明天要做代码审查" |
| 修改状态 | 找到→点击编辑→改状态→保存 | "把代码审查标记完成" |
| 学习成本 | 需了解每个字段 | 自然语言即可 |
| 速度 | 30-60秒 | 5-10秒 |

**不足**：LLM 可能误识别意图、每次操作有 1-3 秒延迟、消耗 API Token、危险操作确认机制弱。

### 5.2 架构基因

```
设计基因 = 个人工具 + 数据安全 + AI 优先 + 本地优先
```

| 基因 | 带来的好处 | 带来的限制 |
|------|-----------|-----------|
| 个人工具 | 简单直接，不需要权限模型 | 难以扩展到协作场景 |
| 数据安全 | Markdown 永远可读 | 三层同步的复杂度 |
| AI 优先 | 交互效率极高 | 依赖 LLM 可用性和成本 |
| 本地优先 | 不依赖云，隐私保护好 | 多端同步困难 |

### 5.3 模块间关系设计（讨论补充）

当前各模块独立运作，缺少模块间的流转设计。应补齐的关系链路：

```
灵感 → 孵化 → 变成笔记（深入研究某个想法）
灵感 → 行动 → 变成任务（马上动手做）
笔记 → 归档 → 归入项目（系统化学习）
任务 → 沉淀 → 产出笔记（做完要总结经验）
项目 → 拆解 → 生成任务（可执行的子项）
笔记/任务 → 提取 → 关联知识图谱（概念节点）

新增的记录类型：
决策 → 为什么选A不选B（决策日志）
反思 → 做完后回头看有什么收获（复盘笔记）
问题 → 遇到了什么困惑（待解决的疑问）
```

### 5.4 多端架构（讨论补充）

```
            ┌─────────────┐
            │  FastAPI 后端 │
            │  (统一 API)   │
            └──────┬───────┘
                   │
         ┌─────────┼─────────┐
         ▼         ▼         ▼
    React Web  Flutter App  MCP 终端
    (现有)     (移动端)    (Claude Code 等)

设计原则：
- 后端 API 是唯一的业务逻辑层，前端只做展示和交互
- 任何新客户端只需对接同一套 API
- MCP Server 作为终端工具的接入层
```

### 5.5 数据边界决策（讨论补充）

```
Import：不做。
  理由：导入的是别人的内容，自己写出来的才是思考。
       AI 让创建变得足够简单（"记个笔记：XXX"），
       不需要做格式转换、数据映射来抢用户。

Export：必须做。
  理由：用户知道随时能拿走数据，才有安全感。
       这是"本地优先"架构的核心承诺。
       支持：单条分享、回顾报告导出、全量备份(zip)。
```

## 六、当前 UI 问题诊断（现状分析）

### 问题一：导航和信息架构 — 6 个平铺页面，没有层次感

```
首页 | 任务 | 灵感 | 笔记 | 项目 | 回顾
  ← 6个平等的入口，但用户 80% 的时间只在首页和任务页
```

- 首页、灵感、笔记、项目本质上都是「查看不同类型的条目」
- 用户需要记住「灵感在第三个、笔记在第四个」
- 侧边栏 6 个入口看起来很多，但信息密度很低
- **根本原因**：按「数据类型」组织页面，而不是按「用户场景」组织

### 问题二：AI 对话是「底层抽屉」，不是核心交互

- 产品最大的差异化是 AI 对话，但 UI 上被压在最底部
- 用户在首页、任务页、笔记页都看到相同的对话面板 — 和当前页面没有关联
- 对话面板遮挡主内容区，拖拽调整高度也是心智负担

### 问题三：首页是「统计报表」，不是「行动中心」

- 用户打开首页看到的是「你有多少数据」，不是「你现在应该做什么」
- 没有 AI 主动建议，没有优先级引导
- 「最近灵感」和「最近记录」是重复信息

### 问题四：列表页功能过于单一

- 每个页面都是一个「只读列表」，没有给用户操作的空间
- 灵感页和项目页几乎是空壳 — 与首页的「最近灵感」重复
- 项目没有进度视图（时间线/看板），只有一个平铺列表

### 问题五：移动端完全不可用

- 侧边栏固定 256px → 移动端直接不可见
- FloatingChat 固定底部 → 移动端遮挡全部内容
- 没有响应式布局

### 问题六：问题反馈是「黑箱」

- 用户提交 bug 后看不到处理进度（提交成功就没了）
- 无法查看自己历史提交的反馈列表
- 没有反馈状态变更通知（处理中/已解决）
- 运营侧通过 MCP log-service 工具管理 issue，但用户侧完全断开

## 七、UI 重设计方案（规划提案）

### 设计原则

| 原则 | 含义 | 当前违反 |
|------|------|---------|
| **对话优先** | AI 是主要交互方式，不是附件 | 对话被压在底部 |
| **场景驱动** | 按用户场景组织，不按数据类型 | 6个平铺页面 |
| **渐进展示** | 默认只看最重要的，按需展开 | 首页堆砌了4个Card |
| **操作就近** | 在哪里看到，就在哪里操作 | 列表页只能看不能改 |

### AI 角色设计：日知

AI 不只是一个对话框，而是一个有名字、有性格的「成长伙伴」。

#### 命名由来

源自顾炎武《日知录》：「日知其所亡，月无忘其所能」— 每天知道一点新东西，每月不忘记已掌握的。《日知录》本身就是「每天记录所思所学」的经典著作，和产品做的事本质相同 — 一个 300 年前用毛笔做，一个今天用 AI 做。

#### 角色设定

| 维度 | 设计 |
|------|------|
| **名字** | 日知 |
| **人设** | 博学但不卖弄的朋友，帮你把思考落到实处 |
| **语气** | 平实、准确、偶尔温厚。不说「亲」「哦~」，也不端着 |
| **自我介绍** | 「你好，我是日知。帮你把每天的想法记下来，日子久了，这些就是你成长的证据。」 |

#### 三种交互模式

| 模式 | 举例 |
|------|------|
| 🎯 教练（主动引导） | 「你这周学了 Rust 但没练手，要不试着写个小项目？」 |
| 🤝 助手（即时响应） | 「好，记下来了。这条灵感要不要展开成笔记？」 |
| 🪞 镜子（反思回顾） | 「你这个月笔记集中在 Rust 和系统设计，方向挺聚焦的。」 |

> 注：日知的完整品牌设计（Slogan、视觉风格等）在第三阶段落地，第一阶段只做对话引导的最小实现。

### 新的信息架构：6 页 → 5 页（分组策略）

```
旧: 首页 | 任务 | 灵感 | 笔记 | 项目 | 回顾
新: 今天 | 探索 | 任务 | 图谱 | 回顾
```

**合并策略**：灵感/笔记/项目合并到「探索」，任务保留独立页面。

| 页面 | 内容 | 合并理由 |
|------|------|---------|
| 今天 | AI 晨报 + 今日任务 + 待处理灵感 | 行动中心 |
| 探索 | 搜索 + 灵感/笔记/项目统一浏览 | 这三个都是「内容型」，浏览逻辑相似 |
| 任务 | 独立任务管理页，保留完整筛选 | 任务是「行动型」，有自己的工作流，不能压缩为 Tab |
| 图谱 | 知识/能力地图 | 核心差异化 |
| 回顾 | 成长报告 | 复盘专用 |
### 新用户 Onboarding：日知对话引导（方案 A）

新用户首次进入「今天」页时，日知主动开口，用真实对话代替教程引导。

```
首次进入 → 今天页 → 日知打招呼：

「你好，我是日知。
 帮你把每天的想法记下来，日子久了，这些就是你成长的证据。

 试试跟我说：
 · "记个灵感：我想学 Rust"
 · "今天要做：完成代码审查"
 · "记个笔记：今天学到..."」

用户第一次对话后，日知自然引导：

「记下来了。你可以在左边的"任务"页看到它。
 要不要再试一个？比如记一条学习笔记。」

用户操作 2-3 次后，引导自然结束。不弹窗、不打勾、不强制走完流程。
```

**为什么选这个方案**：
- 符合 AI-first 产品哲学，Onboarding 就是真实使用
- 日知的角色从第一次见面就开始建立
- B 定位用户不需要示例数据，直接用就懂了
- 最简实现，不需要引导页、示例数据、清除按钮

### 页面 1：今天（Today）— 替代首页

### 问题反馈：从「黑箱提交」到「可追踪闭环」

**当前状态**：右下角浮动按钮 → 填表单 → 提交成功 → 结束。用户完全不知道后续进展。

**新设计**：反馈入口不变，但增加「我的反馈」视图和状态追踪。

```
┌──────────────────────────────────────────────────────────┐
│  入口（保持现有浮动按钮，但增加快捷入口）                     │
│                                                          │
│  1. 任何页面 → 右下角 🐛 按钮 → 打开反馈面板               │
│  2. 探索页搜索栏 → "这个页面有个bug" → AI 自动识别为反馈    │
│  3. 条目详情页 → 操作菜单 → "报告问题"                     │
│                                                          │
│  ┌─── 反馈面板 ──────────────────────────────────────┐    │
│  │  [新建反馈] [我的反馈(3)]                           │    │
│  │                                                    │    │
│  │  新建反馈（现有表单保持不变）：                       │    │
│  │  标题 / 严重程度 / 描述                             │    │
│  │                                                    │    │
│  │  ─────────────────────────────────                 │    │
│  │                                                    │    │
│  │  我的反馈：                                        │    │
│  │  🟡 搜索结果加载缓慢     处理中   2天前             │    │
│  │  🟢 页面布局错位         已解决   1周前             │    │
│  │  🔴 数据丢失风险         待处理   3天前             │    │
│  │                                                    │    │
│  │  点击展开可查看详细状态和处理备注                     │    │
│  └────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘

后端补充：
- GET /feedback/my → 返回当前用户提交的反馈列表（按 reporter 过滤）
- GET /feedback/:id → 返回单条反馈详情（含状态变更历史）
- 可选：状态变更时通过 SSE 推送通知到前端
```

| 设计要点 | 原因 |
|---------|------|
| 保留浮动按钮 | 反馈是全局能力，不应该藏在某个页面里 |
| 增加「我的反馈」Tab | 让用户感知自己的声音被听到了 |
| 多入口上报 | 不同场景有不同上下文，条目详情页上报可自动关联 entry_id |
| AI 自动识别反馈 | "这个页面有个bug" → 意图识别为 feedback → 自动打开反馈面板 |
| 状态追踪 | pending → in_progress → resolved 的可视化，提升用户信任感 |

## 八、新旧对比（现状 vs 规划）

| 维度 | 当前设计 | 新设计 | 为什么更好 |
|------|---------|--------|-----------|
| **页面数量** | 6 个平铺 | 5 个场景（分组） | 减少导航复杂度，任务保留专注体验 |
| **知识图谱** | 隐身功能（需对话触发） | 独立图谱页 + 详情页缩略图 | 核心差异化能力可视化 |
| **AI 位置** | 底部抽屉 | 内嵌每个页面 + 有名字的角色 | 上下文关联，体验连贯 |
| **首页定位** | 统计报表 | 行动中心 | 用户来做事，不是看报表 |
| **灵感出路** | 只能看 | 一键转化 | 解决灵感「只进不出」 |
| **搜索** | 独立页面 | 探索页顶部 | 搜索是高频操作，应该一进入就能用 |
| **详情页** | 纯内容展示 | 关联中心（图谱+条目+AI） | 信息从孤立变联网 |
| **回顾** | 静态数字 | AI 总结 + 曲线 + 知识热力图 | 数据需要解读才有价值 |
| **问题反馈** | 黑箱提交（提交后无跟踪） | 可追踪闭环（我的反馈 + 状态追踪） | 用户信任感 |
| **数据导入** | 不支持 | 不做（用户自己创建即可） | 符合「思考才是成长」的产品哲学 |
| **数据导出** | 不支持 | 支持（单条/报告/全量） | 给用户数据安全感 |
| **移动端** | 不可用 | Flutter 原生 App（录入优先） | 随时随地零摩擦记录 |
| **终端接入** | 不支持 | MCP Server（Claude Code 等可直接操作） | AI 时代的新交互方式 |

## 九、风险与代价

| 风险 | 说明 |
|------|------|
| **合并页面 → 失去专注** | 任务页的筛选能力很强，合并到「探索」后可能不够深入 |
| **AI 内嵌 → 开发量大** | 每个页面都需要独立的 AI 交互逻辑 |
| **减少导航 → 老用户不适应** | 习惯了 6 个入口的用户需要重新学习 |
| **AI 晨报 → 数据不足时尴尬** | 新用户没有数据，晨报不知道说什么 |

## 十、演进路径（三阶段收敛）

> 策略：先补闭环，再改导航，最后做差异化。不改世界观，先让核心链路跑通。

```
━━━ 第一阶段：补闭环（不改世界观）━━━

Phase 0: 新用户 Onboarding（日知对话引导）       ← 影响后续所有功能的感知价值
Phase 1: 首页改版为「今天」                      ← 今日任务 + 待处理灵感 + 未完成项目
Phase 2: 灵感转化流程（→ 任务 / → 笔记）         ← 小改动大价值，解决灵感只进不出
Phase 3: 反馈闭环（我的反馈 + 状态追踪）          ← 小改动，用户信任
Phase 4: 回顾页补趋势对比                        ← 先做基础的周环比/月环比

━━━ 第二阶段：统一探索（减少导航负担）━━━

Phase 5: 探索页（灵感/笔记/项目合并 + 搜索主入口）  ← 搜索成为主入口，类型 Tab 筛选
Phase 6: Export 导出能力                          ← 给用户数据安全感
Phase 7: 条目详情页关联面板                        ← 信息从孤立变联网

━━━ 第三阶段：做真正的差异化 ━━━

Phase 8: 图谱独立页（知识/能力地图可视化）        ← 核心差异化能力
Phase 9: 页面级上下文 AI（日知角色内嵌）          ← 不再是全局大抽屉
Phase 10: 回顾增强（AI 总结 + 成长曲线）          ← 需要数据积累
Phase 11: MCP 终端接入                            ← 技术人群杀手特性
Phase 12: Flutter 移动端（录入优先 MVP）          ← 随时随地零摩擦
Phase 13: AI 晨报/主动推送                        ← 数据积累足够后才有意义
```

关键原则：
- 第一阶段不改信息架构，只在现有 6 页基础上改造首页和补流程
- 第二阶段才开始合并导航，此时用户已经用起来了
- 图谱和 AI 内嵌放到第三阶段 — 这些是锦上添花，不是活下去的前提

## 十二、成功指标体系

### 设计原则

B 定位的小众产品，成功不是「有多少人用」，而是「用的人是不是真的离不开它」。指标不追求量大，追求持续。

### 不看的指标

| 不看 | 原因 |
|------|------|
| 总注册量 | B 定位不追求用户规模 |
| DAU/MAU 比值 | 个人工具不是社交媒体，不需要天天打开 |
| 页面 PV | 看了不代表用了 |
| 平均在线时长 | 长 ≠ 好，高效记录才是目标 |

### 完整指标体系（9 层 27 个）

> ⚠️ 以下「早期基线」是产品初期用来判断方向是否正确的参考值，不是长期目标。随着数据积累再逐步上调。

#### 1. 激活 — 用户有没有度过冷启动

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 首次对话完成率 | 注册后首次与日知对话并成功创建条目的比例 | > 60% |
| 首日创建量 | 注册当天创建的条目数 | ≥ 2 条 |
| 次日返回率 | 第二天再次打开的比例 | > 30% |

#### 2. 习惯 — 用户有没有形成使用节奏

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 周活跃天数 | 一周内有记录的天数 | ≥ 3 天 |
| 周均创建量 | 每周创建的条目数 | ≥ 5 条 |
| 周均对话次数 | 每周与日知对话的次数 | ≥ 3 次 |

#### 3. 价值 — 产品有没有真正帮到用户

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 灵感转化率 | 灵感被转化为任务/笔记的比例 | > 20% |
| 回顾页访问率 | 每月访问回顾页的用户比例 | > 30% |
| 条目被回顾率 | 条目创建后被再次查看/编辑的比例 | > 25% |

#### 4. AI 质量 — 日知听不听得懂、做得好不好

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 意图识别准确率 | 用户说的话被正确分类的比例 | > 70%（初期重点关注错误样本，逐步优化） |
| 对话完成率 | 用户说了一句话后，成功创建了条目的比例 | > 75% |
| 用户重试率 | 同一意图需要说两次以上的比例 | < 25% |
| 平均响应时间 | 从用户发问到日知完成操作的时间 | < 5 秒 |

#### 5. 知识图谱 — 核心差异化有没有落地

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 概念节点数 | 知识图谱中的概念数量 | 持续增长即可 |
| 条目关联率 | 有知识图谱关联的条目占比 | > 30% |
| 图谱页访问率 | 访问图谱页的用户比例 | > 20% |

#### 6. 技术底线

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| API 错误率 | 5xx 错误占请求总量比例 | < 2% |
| 日知响应时间 | P95 响应时间 | < 5 秒 |
| 同步一致性 | SQLite/Markdown 数据一致 | 不丢数据（Neo4j/Qdrant 允许短暂延迟） |

#### 7. 增长 — 口碑和推荐

| 指标 | 定义 |
|------|------|
| NPS / 口碑推荐 | 用户愿不愿意推荐给同事 |
| 邀请转化率 | 被推荐人注册并激活的比例 |
| MCP 使用率 | 通过 MCP 操作的用户比例 |

#### 8. 条目质量 — 数量不等于价值

| 指标 | 定义 |
|------|------|
| 平均条目长度 | 正文内容的平均字数 |
| 笔记关联密度 | 每条笔记平均关联的概念/条目数 |
| 编辑次数 | 条目创建后被编辑的平均次数（编辑多 = 在思考） |

#### 9. 留存

| 指标 | 定义 | 早期基线 |
|------|------|---------|
| 周留存 | 注册后第 4 周仍活跃的比例 | > 20% |
| 月留存 | 注册后第 3 个月仍活跃的比例 | > 10% |

### 第一批实施（P0 + P1 共 10 个）

| 优先级 | 指标 | 数据源 | 理由 |
|--------|------|--------|------|
| **P0** | 首次对话完成率 | 会话记录 | 激活的起点 |
| **P0** | 次日返回率 | 用户活跃时间 | 最基本的留存信号 |
| **P0** | 周活跃天数 | entries 创建时间 | 核心习惯指标 |
| **P0** | 周均创建量 | SQLite entries | 产品使用强度 |
| **P0** | 意图识别准确率 | 会话记录 + 反馈 | AI 体验的根基 |
| **P1** | 灵感转化率 | entries 状态变更 | 闭环是否打通 |
| **P1** | 回顾页访问率 | 页面访问日志 | 成长飞轮是否转起来 |
| **P1** | 条目关联率 | Neo4j 查询 | 知识图谱有没有用 |
| **P1** | 周留存 | 用户活跃时间 | 中期留存 |
| **P1** | API 错误率 | 后端日志 | 技术底线 |

实现方式：不需要独立的分析平台，现有架构即可支持。

## 十三、多端设计

### Flutter 移动端

技术选型：Flutter（跨平台 iOS + Android）

```
设计原则：
- 移动端的核心价值是「随时随地、零摩擦录入」— 打开 → 说一句话 → 关掉，3 秒完成
- 不做「在手机上做复杂操作」
- 后端 API 不需要改，Flutter 只是新的前端客户端
- AI 是一等公民 — 日知对话是核心交互，不是附件
```

#### 导航设计

移动端不是 Web 的缩小版，底部 Tab 只有 3 个：

```
┌─────────────────────────────┐
│                             │
│        (页面内容区)          │
│                             │
├─────────────────────────────┤
│  🏠 今天  │  💬 日知  │  ✅ 任务  │
│            │  (大按钮) │          │
└─────────────────────────────┘
```

探索/图谱/回顾 → 移动端不做独立页面，通过「今天」页入口可跳转 Web 版。

#### MVP 功能清单

| 功能 | 页面 | 优先级 | 说明 |
|------|------|--------|------|
| 登录/注册 | 独立页 | P0 | 复用后端 auth API |

| 快速创建条目 | 日知对话内 | P0 | 对话自动创建任务/灵感/笔记 |

## 十六、总结

### 一句话定义

**先帮用户把东西记下来，再帮用户把东西连起来，最后帮用户看见自己在成长。**

### 当前状态

核心骨架已经落地：统一条目模型、自然语言入口、混合搜索、知识图谱、回顾统计、会话历史、反馈、认证都在代码中实现。但产品深度不足 — 数据模型仍停留在「条目管理系统」（Task/Entry 扩展字段），还没有真正的「目标」「决策」「复盘」「能力」一等实体。更多是一个「能用」的工具，还没到「离不开」的助手。

### 演进策略（务实的三阶段）

**第一阶段：不改世界观，只补闭环**
- 首页改成「今天」— 今日任务 + 待处理灵感 + 最近未完成项目
- 补「灵感转任务/转笔记」
- 补「任务完成后写总结笔记」的轻引导
- 回顾页先补趋势对比，不急着上复杂成长曲线

**第二阶段：统一探索，减少导航负担**
- 灵感、笔记、项目合并浏览到「探索」页
- 任务保持独立工作区
- 搜索成为探索页的主入口

**第三阶段：做真正的差异化**
- 图谱独立页
- 条目详情页的关联面板
- 页面级上下文 AI（日知角色内嵌）
- AI 晨报/主动推送（数据积累足够后）
