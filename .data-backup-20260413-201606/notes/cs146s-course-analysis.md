# CS146S: The Modern Software Developer 课程设计深度解析

> 斯坦福大学 CS146S 课程分析，探讨 AI 时代软件开发方式的转型。

## 一、课程定位

**CS146S: The Modern Software Developer**（现代软件开发者）

**核心理念**：AI 正在改变软件开发的方式，现代开发者需要学会与 AI 协作。

这不是"AI 应用开发课"，而是"**软件开发方式转型课**"。

---

## 二、每周内容详解

### Week 1: Prompting Techniques

**学习内容**：
- 使用 Ollama 本地运行 LLM（mistral-nemo, llama3.1）
- 练习 6 种技术：K-shot、CoT、Tool Calling、Self-consistency、RAG、Reflexion
- 每种技术有具体任务和测试脚本，迭代直到测试通过

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 沟通能力 | 学会"说 AI 能理解的话" |
| 理解边界 | 知道 LLM 能做什么、不能做什么 |
| 问题分解 | 把复杂任务拆成 AI 能处理的步骤 |

**产出**：6 个 prompt 练习文件

---

### Week 2: Action Item Extractor

**学习内容**：
- 给你一个**已存在的** FastAPI + SQLite 应用
- 用 **Cursor**（AI 编程工具）完成 5 个 TODO

**5 个 TODO**：

| TODO | 内容 | 锻炼能力 |
|------|------|---------|
| TODO 1 | 用 LLM 实现 `extract_action_items_llm()` | Prompt + 结构化输出 |
| TODO 2 | 写单元测试 | 测试 AI 代码 |
| TODO 3 | 重构代码（API schemas、数据库层、错误处理） | 代码质量把控 |
| TODO 4 | 添加新端点 + 前端按钮 | 全栈扩展 |
| TODO 5 | 用 Cursor 生成 README | 文档自动化 |

**设计意图**：

```
功能实现 → 测试 → 重构 → 扩展 → 文档
```

不只是"加个 LLM"，而是完整的工程实践闭环。

**产出**：扩展后的应用 + 单元测试 + README

---

### Week 3: MCP Server

**学习内容**：
- 选择一个外部 API（天气、GitHub、Notion 等）
- 实现 MCP Server，暴露至少 2 个 Tools
- 处理错误、超时、限流
- 选择部署模式：本地（STDIO）或远程（HTTP）

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 协议理解 | 理解 AI Agent 的工具调用机制 |
| 接口设计 | 设计 AI 能正确调用的 API |
| 可靠性 | 错误处理、超时、限流 |
| 可选：认证 | API Key 或 OAuth2 |

**产出**：可运行的 MCP Server + README

---

### Week 4: Autonomous Coding Agent (Claude Code)

**学习内容**：
- 使用 **Claude Code** 构建至少 2 个自动化
- 在同一个 FastAPI playground 上实验

**自动化选项**：

| 类型 | 内容 |
|------|------|
| Slash commands | 可复用的工作流（如 `/tests`、`/docs-sync`） |
| CLAUDE.md | 项目级别的 AI 指导文件 |
| SubAgents | 多 Agent 协作（TestAgent + CodeAgent） |

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 工作流设计 | 把重复任务自动化 |
| 上下文管理 | 用 CLAUDE.md 给 AI 提供项目上下文 |
| Agent 协作 | 设计多个 Agent 的分工 |

**产出**：2+ 自动化 + writeup

---

### Week 5: Agentic Development with Warp

**学习内容**：
- 使用 **Warp**（AI 终端）
- 在**同一个 FastAPI playground** 上实验（与 Week 4 相同应用）
- 对比不同工具的工作流

**自动化选项**：

| 类型 | 内容 |
|------|------|
| Warp Drive | 保存 prompts、rules、MCP servers |
| 多 Agent 并发 | 多个 Warp tab 同时处理独立任务 |

**关键技术**：
- git worktree：避免多个 Agent 互相覆盖

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 并发协调 | 让多个 AI 同时工作 |
| 资源管理 | 隔离工作区 |
| 工具对比 | 体验不同工具的优劣 |

**产出**：2+ 自动化 + 多 Agent 协作记录

---

### Week 6: Security with Semgrep

**学习内容**：
- 使用 **Semgrep** 扫描代码漏洞
- 修复至少 3 个安全问题
- 覆盖三类漏洞：
  - SAST：代码漏洞（SQL 注入、XSS 等）
  - Secrets：密钥泄露
  - SCA：依赖漏洞

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 安全意识 | 识别 AI 代码中的安全风险 |
| 工具使用 | 使用静态分析工具 |
| 修复能力 | 知道如何正确修复漏洞 |

**产出**：漏洞报告 + 3 个修复

---

### Week 7: AI Code Review with Graphite

**学习内容**：
- 用 AI 工具实现 4 个任务
- **每个任务用 1-shot prompt**（一次性说清楚需求）
- 用 **Graphite Diamond** 生成 AI 代码审查
- 对比：你的审查 vs AI 的审查

**流程**：

```
创建分支 → 1-shot prompt 实现 → 手动审查 → 开 PR → AI 审查 → 对比反思
```

**1-shot prompt 的含义**：

> 一次性把需求说清楚，不给 AI 第二次机会追问。这是对描述能力的硬性考核。

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 需求描述 | 一次说清楚，不依赖迭代 |
| 代码审查 | 逐行审查 AI 生成的代码 |
| 判断能力 | 知道 AI 审查何时可信、何时不可信 |

**产出**：4 个 PR + 审查对比反思

---

### Week 8: Multi-Stack Build

**学习内容**：
- 用 **3 种技术栈**实现同一个应用
- 至少一个用 **bolt.new**（AI 应用生成平台）
- 至少一个用非 JavaScript 语言（Django、Rails 等）
- 鼓励部署

**应用要求**：
- CRUD 操作
- 持久化存储
- 基本验证和错误处理
- 功能性 UI
- 运行说明（+ 部署链接如果部署）

**锻炼能力**：

| 能力 | 说明 |
|------|------|
| 技术栈无关 | 不被特定框架绑定 |
| AI 生成管理 | 管理 AI 生成的代码库 |
| 快速交付 | 用 AI 快速构建完整应用 |

**产出**：3 个完整应用 + README + 部署链接（可选）

## 三、课程设计逻辑

### 阶段划分

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：理解 AI（Week 1-3）                                  │
│  ├── Week 1: 与 AI 沟通（6 种 Prompt 技术）                   │
│  ├── Week 2: 工程实践（5 个 TODO 完整闭环）                    │
│  └── Week 3: 能力扩展（MCP Server）                           │
├─────────────────────────────────────────────────────────────┤
│  阶段二：自动化工作流（Week 4-5）                              │
│  ├── Week 4: Claude Code（同一 playground）                   │
│  └── Week 5: Warp 并发 Agent（同一 playground）               │
├─────────────────────────────────────────────────────────────┤
│  阶段三：质量把控（Week 6-7）                                  │
│  ├── Week 6: 安全审查（Semgrep）                              │
│  └── Week 7: 代码审查（1-shot + Graphite）                    │
├─────────────────────────────────────────────────────────────┤
│  阶段四：综合验证（Week 8）                                    │
│  └── Week 8: 3 技术栈 + 鼓励部署                               │
└─────────────────────────────────────────────────────────────┘
```

### 设计原则

**1. 从使用者到构建者**

```
Week 1: 用 AI（理解能力）
Week 2: 集成 AI（工程实践）
Week 3: 给 AI 造工具（扩展能力）
Week 4-5: 让 AI 自动化工作（工作流设计）
```

**2. 完整工程闭环**

Week 2 的 5 个 TODO 体现了这一点：

```
功能 → 测试 → 重构 → 扩展 → 文档
```

**3. 工具对比学习**

Week 4 & Week 5 用同一个 playground：
- 排除"理解业务"的变量
- 专注对比不同工具的工作流
- 培养"工具选择"的决策能力

**4. 强调"审查"而非"编写"**

Week 6 + Week 7 占了课程的 1/4：
- 安全审查：发现 AI 代码的漏洞
- 代码审查：判断 AI 代码对不对

**5. 硬性考核描述能力**

Week 7 的 1-shot prompt 要求：

> 一次性说清楚需求，是对描述能力的硬性考核，不是软技能。

**6. 鼓励真实交付**

Week 8 鼓励部署：

> AI 生成的代码要能真正交付，不是 demo。

## 五、核心能力模型

### 技能价值变化

| 传统开发 | AI 时代开发 |
|---------|------------|
| 记住 API 语法 | 描述清楚需求 |
| 手写代码 | 审查 AI 代码 |
| 单一技术栈 | 快速跨栈 |
| 单打独斗 | 指挥 AI 协作 |
| 写代码 | 设计 + 审查 |

### 现代软件开发者能力模型

```
┌─────────────────────────────────────────┐
│         现代软件开发者能力模型            │
├─────────────────────────────────────────┤
│  描述能力：一次说清楚需求（1-shot）        │
│  分解能力：把复杂问题拆成 AI 能处理的步骤   │
│  架构能力：理解系统整体结构               │
│  审查能力：判断 AI 代码对不对             │
│  安全意识：发现 AI 代码的风险             │
│  工具链：熟练使用 AI 开发工具             │
│  工程闭环：功能→测试→重构→扩展→文档       │
└─────────────────────────────────────────┘
```

## 六、课程不教什么

| 不教的内容 | 原因 |
|-----------|------|
| 框架深入知识 | AI 帮你写，你需要会审查 |
| 算法实现细节 | AI 帮你写，你需要会判断 |
| 性能优化技巧 | AI 帮你写，你需要会验证 |
| 特定语言高级特性 | 跨栈能力比深钻某一门语言更重要 |

## 七、核心洞察

**一句话总结**：

> **以前比谁写得快，现在比谁看得准。**

**课程设计的本质**：

1. **理解 AI**：知道它的能力和边界
2. **指挥 AI**：学会描述需求、分解任务
3. **审查 AI**：判断代码对不对、安不安全
4. **超越技术栈**：不被特定框架绑定
5. **工程闭环**：不只是写代码，而是完整交付

---

## 八、总结

CS146S 课程揭示了一个重要趋势：

**Vibe Coding** 的真正含义——不是"用 AI 写代码"，而是"**重新定义开发者的工作方式**"。

从执行者变成指挥官，从写代码变成审查代码，从单一技术栈变成快速跨栈。这才是 AI 时代开发者的核心竞争力。
