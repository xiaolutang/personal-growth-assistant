## 实现细节

### Tasks 页面
- 每个 TaskCard 被 Dismissible 包裹（`key: ValueKey('dismissible_${entry.id}')`）
- **左滑 (startToEnd)**: 绿色背景 + Icons.check -> 执行完成操作
- **右滑 (endToStart)**: 红色背景 + Icons.delete -> 执行删除操作
- 已完成任务左滑完成时 `confirmDismiss` 返回 false 并提示
- ReorderableListView 内的 Dismissible 通过 KeyedSubtree + Dismissible 嵌套实现，拖拽（长按）和滑动（水平手势）不冲突

### Inbox 页面
- 每个列表项被 Dismissible 包裹（`key: ValueKey('dismissible_${entry.id}')`）
- **只支持左滑 (startToEnd)**: 红色背景 + Icons.delete -> 执行删除操作
- `direction: DismissDirection.startToEnd` 限制方向

### 乐观更新策略
1. **完成操作**: Dismissible onDismissed 触发 -> 从本地 `_reorderedEntries` 移除 -> 调用 `updateEntryStatus(entryId, 'complete')` -> 成功则刷新列表 + SnackBar 带「撤销」 -> 失败则回滚到原位 + 错误提示
2. **删除操作**: Dismissible onDismissed 触发 -> provider 层 `deleteEntry` 乐观移除 -> 成功则 SnackBar 带「撤销」 -> 失败则 provider 层自动回滚到原位
3. **撤销**: SnackBar action 点击 -> 调用 `restoreEntry(entry, originalIndex)` 恢复到原位

### Provider 新增方法
- `EntryListNotifier.deleteEntry(entryId)`: 乐观删除，记录原位索引，失败时插回
- `EntryListNotifier.restoreEntry(entry, originalIndex)`: 撤销删除，恢复到原位
- `InboxNotifier.deleteEntry(entryId)`: 同上
- `InboxNotifier.restoreEntry(entry, originalIndex)`: 同上

- [x] 使用 `ValueKey('dismissible_${entry.id}')`，entry.id 为 UUID，全局唯一
- [x] 与 ReorderableListView 的 `ValueKey(entry.id)` 不冲突（不同前缀）

### 乐观更新正确性
- [x] Tasks 完成操作：先从本地 `_reorderedEntries` 移除，API 失败时 `_rollbackEntry` 插回原位
- [x] Tasks 删除操作：通过 provider 层乐观删除，provider 内部自动回滚
- [x] Inbox 删除操作：通过 provider 层乐观删除，provider 内部自动回滚
- [x] 撤销：通过 `restoreEntry` 恢复到记录的 originalIndex 位置

### 内存泄漏
- [x] `mounted` 检查在所有 async 回调中执行
- [x] 无 StreamController 或其他需要 dispose 的资源
- [x] SnackBar 通过 ScaffoldMessenger 管理，不持有页面引用

### API 错误处理
- [x] Provider 层 deleteEntry：try-catch 包裹，失败时回滚 entries 并设置 error
- [x] Page 层：API 失败时显示错误 SnackBar
- [x] 完成操作失败：回滚本地列表 + 错误提示
- [x] 删除操作失败：provider 自动回滚 + 错误提示
