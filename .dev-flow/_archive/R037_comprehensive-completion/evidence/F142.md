# F142: 离线更新/删除拦截

## 任务描述
离线时更新操作拦截并加入 offlineSync 队列，离线时删除操作拦截并加入队列，队列条目包含操作类型（update/delete），上线后队列自动回放，回放失败重试最多 3 次。vitest 覆盖拦截和回放场景。

## 实现变更

### 1. offlineSync.ts — 新增队列优化（`optimizePendingQueue`）

**文件**: `frontend/src/lib/offlineSync.ts`

新增导出函数 `optimizePendingQueue`，在 sync() 回放前优化 pending 队列：

- **同条目多次 PUT 合并**: 累积合并 body（partial patch 语义），保留所有字段，避免用户数据丢失
- **同条目 PUT→DELETE 冲突**: 丢弃所有 PUT，只保留最后一个 DELETE
- **多条 DELETE 合并**: 保留最后一个 DELETE
- **POST 条目不参与优化**
- 合并后被丢弃的队列项从 IDB 中删除

### 2. 现有功能（已实现，无需修改）

以下功能在之前的迭代中已实现：

- **离线 update 拦截入队**: `taskStore.updateEntry` 中 `!navigator.onLine` 分支
- **离线 delete 拦截入队**: `taskStore.deleteTask` 中 `!navigator.onLine` 分支
- **5xx 重试最多 3 次**: `offlineSync.sync` 中 retry_count > 3 逻辑
- **上线后自动回放**: `offlineSync` 中 online 事件监听 + initSync()
- **操作类型区分**: 队列项 method 字段（POST/PUT/DELETE）

## 测试覆盖

**文件**: `frontend/src/lib/__tests__/offlineSync.update-delete.test.ts` (20 个测试)

覆盖所有 8 个 test_tasks：

| test_task | 测试用例 |
|-----------|---------|
| 离线 update 拦截入队 | 离线时 PUT 操作入队并在回放时调用 updateEntry API |
| 离线 delete 拦截入队 | 离线时 DELETE 操作入队并在回放时调用 deleteEntry API |
| 5xx 重试最多 3 次 | 5xx 错误递增 retry_count，达到 3 次后标记 failed |
| 超时重试 | 网络超时（无 status）递增 retry_count；多次超时后最终达到上限标记 failed |
| 快速切换在线/离线 | 同步中 online 事件触发不会导致重入 |
| 同条目多操作冲突(update→delete) | 同一 entry 的 PUT + DELETE 合并后只执行 DELETE |
| 同条目多次 update 合并 | 同一 entry 的多次 PUT 累积合并 body（partial patch 语义）；partial patch 合并：不同字段的更新全部保留 |
| 上线后自动回放 | 混合 PUT 和 DELETE 队列全部回放成功；部分回放失败不影响其他操作 |

额外测试（`optimizePendingQueue` 纯函数）：
- 空数组返回空优化列表
- POST 条目不参与合并
- 单条 PUT 不合并
- 单条 DELETE 不受影响
- 多条 DELETE 保留最后一条
- optimizePendingQueue 累积合并多次 PUT 的 body
- optimizePendingQueue 正确处理 update→delete 冲突
